## Plan: Food Picker Priority-1 Full Execution Plan

此計畫以最小風險達成你已確認需求：前端弱化 AI 摘要、隱藏品項欄位、短評/備註不受 visited 控制、未造訪卡片灰色主題、release 去硬編碼。資料庫先不做破壞性遷移，優先透過前後端流程重構交付。

**Objectives**
1. 短評/備註欄位在新增與編輯流程永遠可見，visited 不再控制其可見性。
2. 品項 items 在前端不可見，但資料層保留以確保舊資料與推薦流程相容。
3. AI 摘要由後端自動生成，使用者不可手動產生/修改。
4. AI 摘要在清單與表單隱藏，僅編輯頁以唯讀方式弱化呈現。
5. AI 摘要生成需納入是否造訪狀態。
6. release 改為自動讀取 docs/releases，不再硬編碼 import。
7. 未造訪卡片整張套用灰色主題（bg #F5F5F5 / border #E0E0E0 / text #7F7F7F）。

**Out of Scope**
1. 不刪除 restaurants.items 欄位，不做 DB migration 回填。
2. 不新增 release 後台管理（CMS/DB）。
3. 不在本輪引入推薦演算法大改（僅必要提示文案微調）。

**Architecture and Data Flow Decisions**
1. UI 隱藏不等於資料移除：items 仍由後端保留，前端不呈現輸入與展示。
2. 摘要責任集中到 server：POST/PATCH 由 API route 決定何時重算與寫回 ai_summary。
3. 摘要失敗不阻斷主流程：餐廳更新成功優先，摘要採 best-effort。
4. release 採 build/runtime-safe 的檔案自動載入策略，避免每次新增 release 需改程式碼。

**Execution Phases**
1. Phase 0 - Baseline and safeguards
1. 確認 Next.js 16 + webpack 模式下 release markdown 載入方案可用，避免 Cloudflare/OpenNext 不相容。
1. 對齊 Restaurant type 與 API payload 欄位，建立變更清單與回歸檢查表。
1. 定義摘要重算欄位集合：review、tags、visited（可含 items 作相容性判斷）。

2. Phase 1 - Form and UI semantics
1. 調整 RestaurantForm：短評/備註欄位固定顯示，不再受 visited 開關控制。
1. visited 只影響造訪後欄位（如 rating）顯示，不影響短評欄位。
1. 隱藏 items 前端輸入區塊與展示區塊，保留 state 映射相容。
1. 保證切換 visited 不清空短評內容，避免資料遺失體驗。

3. Phase 2 - AI summary policy (server-owned)
1. 移除前端手動摘要按鈕與可編輯輸入。
1. 建立頁不顯示摘要欄位；清單頁不顯示摘要；編輯頁可顯示唯讀摘要資訊卡。
1. POST /api/restaurants：建立資料後自動生成摘要並回寫。
1. PATCH /api/restaurants/[id]：偵測關鍵欄位變更後自動重算摘要。
1. generate-summary prompt 納入 visited 狀態，確保摘要語氣與資訊一致。
1. 摘要失敗時記錄錯誤，主 API 回傳成功且保留舊摘要或 null。

4. Phase 3 - Release auto-loading
1. 重構 release 載入模組為自動收集 docs/releases 下 markdown。
1. 以檔名規則與日期排序決定顯示順序。
1. 保留顯示筆數常數化設定（預設 3，可快速切全量）。
1. 補上 parser fallback（title/date 缺失時仍可顯示基本資訊）。

5. Phase 4 - Card theming and readability
1. 在 RestaurantCard 套用未造訪灰色整卡樣式。
1. 驗證推薦頁選取框與灰色主題同時存在時仍清楚。
1. 調整 badge/tag/button 對比，確保可讀性與操作辨識。

6. Phase 5 - Regression hardening
1. 跨頁面確認欄位語意一致（新增、編輯、列表、推薦）。
1. 確認隱藏 items 後不破壞送出 payload 與資料回填。
1. 確認摘要自動重算不造成明顯延遲或連續錯誤。

**API Contract Adjustments**
1. POST /api/restaurants
1. 請求維持相容，回應包含 server 生成後的 ai_summary（成功時）。
1. 若摘要失敗，主體仍成功，ai_summary 允許保留舊值或為 null。
2. PATCH /api/restaurants/[id]
1. 當 review/tags/visited（可含 items）變更，server 自動重算 ai_summary。
1. 忽略 client 端手動傳入 ai_summary 覆蓋（或由 server 覆寫）。
3. POST /api/restaurants/generate-summary
1. 內部仍可復用，但前端不提供手動入口。

**Verification Matrix**
1. Static checks
1. npx tsc --noEmit 應全綠。
1. npm run build 應成功（webpack）。
2. Functional checks
1. 新增未造訪餐廳：短評可填、items 前端不可見、摘要自動生成。
1. 編輯切換 visited：短評不隱藏不清空，摘要內容因 visited 改變而更新。
1. 清單頁：摘要不可見、未造訪卡片灰色主題正確。
1. 編輯頁：摘要可唯讀查看但不可修改。
1. release：新增 markdown 後首頁自動出現。
3. Regression checks
1. 推薦功能仍可回傳結果。
1. 既有資料含 items 的餐廳可正常顯示與編輯。

**Definition of Done**
1. 需求五點全部可被手動驗證重現。
2. 無型別錯誤、可成功 build。
3. 不需 DB migration 即可上線。
4. 新增 release 不再需要改 import 程式碼。

**Risk and Mitigation**
1. 風險：摘要生成增加 API 延遲。
1. 緩解：摘要失敗不阻斷主流程，必要時可改為背景重試。
2. 風險：灰色卡片造成按鈕對比不足。
1. 緩解：局部覆蓋按鈕/badge 前景色，保留可讀性。
3. 風險：release 自動讀檔在部署環境行為差異。
1. 緩解：優先採與現行 webpack build 相容方案並以 npm run build 驗證。

**Rollback Strategy**
1. UI rollback
1. 先保留舊 props 與條件渲染入口，必要時可快速恢復摘要顯示或卡片原色。
2. API rollback
1. 摘要自動重算邏輯以條件區塊包裹，出現異常可暫時停用自動重算而不影響 CRUD。
3. Release rollback
1. 保留 fallback 靜態列表機制或快速切回原讀取模式。

**Decisions Locked**

3. 品項欄位前端不可見，資料層保留。
4. 短評/備註永遠可見，不因 visited 隱藏。
5. 摘要生成納入是否造訪。
